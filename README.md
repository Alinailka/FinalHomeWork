# 1. Перечень автоматизируемых сценариев

## Тест-кейс "Переход на страницу профессии ["Тестировщик ПО"](https://netology.ru/programs/qa#/) с главной страницы сайта ["Нетология"](https://netology.ru/)"

>  ###Сценарий 1.
> Переход на страницы профессии нажатием на кнопку "Каталог курсов" в хедере: "Каталог курсов" --> Программирование --> Тестировщик ПО
> Ожидаемый результат: Откроется страница профессии "Тестировщик ПО"

>  ###Сценарий 2.
> Переход на страницу профессии выбором в блоке направлений обучения "Программирование": Программирование --> Тестировщик ПО
> Ожидаемый результат: Откроется страница профессии "Тестировщик ПО"

>  ###Сценарий 3.
> Переход на страницу профессии нажатием на кнопку "Выбрать курс": "Выбрать курс" --> Тестировщик ПО 
> Ожидаемый результат: Откроется страница профессии "Тестировщик ПО"

>  ###Сценарий 4.
> Переход на страницу профессии нажатием на кнопку "Каталог курсов" в футере: "Каталог курсов" --> Тестировщик ПО
> Ожидаемый результат: Откроется страница профессии "Тестировщик ПО"

## Тест-кейс "Переход к [формы записи](https://netology.ru/programs/qa#/order) на курс на странице профессии ["Тестировщик ПО"](https://netology.ru/programs/qa#/)"  

>  ###Сценарий 1. 
> Переход к форме записи нажатием на кнопку "Записаться" в начале страницы профессии
> Ожидаемый результат: Страница профессии перелистывается до формы записи 

>  ###Сценарий 2. 
> Перейти к форме записи пролистыванием страницы профессии до конца
> Ожидаемый результат: Пользователь видит не заполненную форму записи

>  ###Сценарий 3. 
> Перейти к форме записи нажатием на кнопку "Записаться" во всплывающем модуле вверху страницы профессии
> Ожидаемый результат: Страница профессии перелистывается до формы записи

## Тест-кейс "Заполнение [формы записи](https://netology.ru/programs/qa#/order) на курс ["Тестировщик ПО"](https://netology.ru/programs/qa#/)"

>  ###Сценарий 1. 
> Заполнение формы записи валидными данными (Имя: Алина, Номер телефона: 79996668855)
> Ожидаемый результат: Пользователь видит во всплывающем модуле сообщение "Поздравляем! Вы успешно записались на курс "Тестировщик ПО". Скоро мы свяжемся с вами и ответим на все интересующие вопросы"

>  ###Сценарий 2. 
> В форме записи поле "Имя" не заполнено (Имя: Поле не заполнено, Номер телефона: 79996668855)
> Ожидаемый результат: Поле "Имя" выделено красным цветом и сообщением об ошибке "Обязательное поле"

>  ###Сценарий 3. 
> В форме записи поле "Номер телефона" не заполнено (Имя: Алина, Номер телефона: Поле не заполнено)
> Ожидаемый результат: Поле "Номер телефона" выделено красным цветом и сообщением об ошибке "Обязательное поле"

>  ###Сценарий 4.
> В форме записи поле "Номер телефона" заполнено невалидными данными (Имя: Алина, Номер телефона: 589632)
> Ожидаемый результат: Поле "Номер телефона" выделено красным цветом и сообщением об ошибке "Номер в формате +9 (999) 999-99-99"

# 2. Перечень используемых инструментов с обоснованием выбора				

## 2.1 JUnit 5	
> Фреймворк для модульного тестирования программного обеспечения на языке Java.
## 2.2 Selenide	
> Фреймворк для автоматизированного тестирования, позволяющий эффективно работать с веб-элементами и писать стабильные тесты
## 2.3 Docker	
> Програмное обеспечение, позволяющее создавать любую среду тестирования на одной машине
## 2.4 Java	
> Объектно-ориентированный язык программирования для написания автотестов
## 2.5 GitHub	
> Система контроля версий, дающая возможность совместной разработки и позволяющая хранить код в течение всего жизненного цикла
## 2.6 Allure	
> Фреймворк для сбора данных и построения отчетов о результатах тетсирования
## 2.7 Faker	
> Библиотека, генерирующая фейковые данные пользователей
## 2.8 Appveyor/Github Action	
> Веб-сервис непрерывной интеграции, предназначенный для сборки и тестирования программного обеспечения расположенного на GitHub

# 4. Перечень и описание возможных рисков при автоматизации

## 4.1 Изменение кода объекта тестирования	
> Если сайт будет полностью переписан,то и тесты должны быть написаны заново	
## 4.2 Изменения в дизайне объекта тестирования	
> Перемещение элементов оформления может повлечь за собой падение тестов 	
## 4.3 Потребность в поддержке тестов в рабочем состоянии, выход релиза может задерживаться	
> Автотесты могут падать по причине багов тестируемого объекта и дефекта в самих тестах. В случае обнаружения дефектов в тестах, их необходимо исправлять	
## 4.4 Трудозатраты на автоматизацию слишком высоки, стоимость тестирования возрастает	
> Для написания и поддержания автотестов требуется сотрудник со знаниями программирования, который стоит дороже. Написание автотестов занимает много времени, которое впоследствие может окупиться, а может и нет, если объект тетстирования динамично меняется или сложен по своей структуре	
## 4.5 Сложности при поиске локаторов веб-элементов
> На страницах сайтов со сложной структурой могут быть расположены элементы не имеющие явной привязки, найти локатор для написания автотестов может быть затруднительным.
